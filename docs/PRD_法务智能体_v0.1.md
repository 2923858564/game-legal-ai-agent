## 1. 产品概述

### 1.1 项目背景

传奇游戏公司法务部日常处理以下业务场景：

- **合同管理**：推广合作、联运、外包、采购等合同的预审→终审→归档全流程
- **诉讼管理**：传奇IP侵权（私服/外挂/授权链）、合同纠纷、劳动仲裁等案件的跟踪与节点管理
- **员工咨询**：工伤处理、纠纷建议、法律知识查询等跨部门法务需求
- **广告合规**：游戏推广素材的合规审查（极限词、虚假宣传、概率公示等）
- **客诉处理**：玩家退款、欺诈举报、未成年人消费等客诉的法务支持

当前流程依赖法务人员手工处理，存在以下痛点：

1. 合同审批完成后，归档字段需人工整理至表格，耗时且易遗漏
2. 法务重复回答同类咨询（工伤流程、合同预审基础问题）
3. 诉讼案件节点分散在个人记录中，缺乏统一跟踪
4. 员工侧缺乏自助法务工具，小事也需@法务

### 1.2 产品目标

| 目标 | 描述 |
|------|------|
| **合同资产可查可统计** | 审批完成后自动归档至多维表格，结构化可检索 |
| **对内核心能力可用** | 终审、诉讼管理、文书起草等法务专用能力可稳定输出 |
| **对员工自助可用** | 合同预审、工伤指引、FAQ 等员工可自助完成，复杂场景转法务 |
| **交付可运营可迭代** | 知识库模块化、提示词可维护、法务人员可上手配置 |

### 1.3 建设原则

1. **全程飞书体系**：优先使用飞书原生能力（多维表格、Aily、审批连接器），不单独开发网页版
2. **双智能体架构**：内部法务智能体（A）+ 员工自助智能体（B），权限和能力隔离
3. **输出可校验**：所有法律分析输出必须符合引用规范（法可依 + 参考例文原文引用）
4. **渐进交付**：Claude Code 原型验证 → 飞书Bot 内测 → Aily 正式上线

## 3. 功能需求

### 3.1 模块 M1：合同审批自动归档

| 需求 ID | 描述 | 优先级 |
|---------|------|--------|
| M1-01 | 审批实例完成后，自动将关键字段写入多维表格 | P0 |
| M1-02 | 归档字段包括：合同编号、名称、类型、签约主体、有效期、金额、发起人、审批结论、附件链接 | P0 |
| M1-03 | 支持定时 API 拉取作为备选触发方式 | P1 |
| M1-04 | 历史存量合同可按需批量导入 | P2 |
| M1-05 | 权限控制：法务/管理员可编辑全部记录；普通员工仅查看本人提交 | P1 |

**技术路径**：飞书审批连接器事件触发 → 多维表格写入。实例以企业开通能力为准。

### 3.2 模块 M2：智能体 A（内部法务）

#### A1 合同终审

| 需求 ID | 描述 | 优先级 |
|---------|------|--------|
| A1-01 | 支持上传全格式合同 docx/pdf 或粘贴文本 | P0 |
| A1-02 | 输出：风险清单（四级分级）、修改建议、条款漏洞、错别字与逻辑矛盾 | P0 |
| A1-03 | 审核模式：快速（核心条款30min）/ 标准（全面2h） | P1 |
| A1-04 | 终审输出包含：法律审核意见书 + 批注版合同 | P1 |

**审核框架**：

2. 四级风险：🔴致命 / 🟠重要 / 🟡一般 / 🔵轻微

| 需求 ID | 描述 | 优先级 |
|---------|------|--------|
| B3-01 | 端到端工伤指引：是否属工伤→申报时限→赔偿项目→证据清单→鉴定流程 | P0 |

5. 劳动能力鉴定流程

```

### 3.4 模块 M4：横切能力

#### 引用规范（强制约束，所有智能体通用）

所有法律分析输出必须遵循以下规范（已原型验证通过）：

```
1. 法可依原则：每条建议必须有法律依据引用
   - 格式：《法律名称》第X条 / 法释〔20XX〕XX号
   - 无法条标注格式：【法律依据】（网上搜索：{来源简述}）
   - 完全未查到格式：【法律依据】（未查到直接引用的法例）

2. 参考例文规则：凡是输出建议/方案/话术，必须附参考例文原文
   - 原文引用，不得改写概括
   - 标注参考来源出处

3. 自检清单：输出前检查
   [ ] 每条建议是否有法律依据？
   [ ] 参考例文是否原文引用？
   [ ] 引用来源是否标注？
   [ ] 法条是否现行有效？
   [ ] 法规名称+条文号是否准确？
```

#### 条款组织规范

合同起草/修订输出时，按业务场景分条，不按法律概念拆分（已原型验证通过）：

```
正例：第三条 下架处理
      3.1 责任归属 → 3.2 处理方式 → 3.3 对应费用
反例：第四条 费用（只写钱）... 第七条 责任（读者来回翻）
```

## 4. 非功能需求

### 4.1 性能

| 需求 | 指标 |
|------|------|
| 合同终审响应 | 标准模式 ≤ 2min |
| 多合同对比响应 | 2~3份对比 ≤ 3min |
| 知识库 FAQ 响应 | ≤ 10s |
| B 智能体每日配额 | 建议每人 20 次/日（最终以成本评估为准） |

### 4.2 安全

| 需求 | 描述 |
|------|------|
| 权限隔离 | A/B 智能体数据和能力严格隔离 |
| 对外输出控制 | B 智能体产出文书需法务二次审核后对外 |
| 敏感信息保护 | 合同内容、案件信息不进入公共知识库 |
| 会话数据 | 保留策略符合公司数据治理要求 |

### 4.3 合规

| 需求 | 描述 |
|------|------|
| 法条时效性 | 输出前自动校验法律版本（三阶段·Phase 2） |
| 免责声明 | B 智能体所有输出附带"非法务正式意见"标识 |
| 游戏行业分区 | 传奇IP/版权/广告法等场景知识库独立分区，不与一般民事模板混用 |

阶段二：飞书Bot 内测（当前阶段）
  环境：cc-connect + Claude Code
  交付物：feishu_handler.py + relay.py（含白名单+管理员）
  测试范围：法务团队内部使用

阶段三：飞书 Aily 正式上线
  环境：飞书 Aily 平台
  迁移内容：
    智能体 A → Claude Code（推理引擎）+ Aily（对话前端）
    智能体 B → Aily（纯云端，简化版 skill）
    合同归档 → 审批连接器 + 多维表格
```

### 6.3 模型选型考量

| 模型 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| Claude (当前方案) | 法律推理精度高、法条引用准确 | 需本地部署/API 费用 | A 智能体（核心法务） |
| Aily 内置模型 | 云端免运维、与飞书原生集成 | 法律精度待验证 | B 智能体（员工侧） |

**建议**：输出对比评估表（能力、国内法务适配、年/月成本）后再做最终决定。

## 8. 验收标准

### 8.1 合同归档验收

| 验收项 | 标准 |
|--------|------|
| 审批完成触发 | 审批实例完成后 5 分钟内归档写入 |
| 字段完整率 | ≥ 99%（必填字段全部写入） |
| 权限控制 | 员工仅见本人记录，法务见全部 |

### 8.2 智能体 A 验收

| 验收项 | 标准 |
|--------|------|
| 合同终审准确性 | 风险识别准确率 ≥ 85%（以法务复核为准） |
| 多合同对比 | 2~3 份合同 3 分钟内输出对比表 |
| 文书起草 | 格式正确、法条引用准确、可直接打印 |
| 引用规范符合率 | 100%（每份输出检查法可依 + 参考例文） |
| 法律校验 | 发现过期法条并提示更新的召回率 ≥ 90% |

### 8.3 智能体 B 验收

| 验收项 | 标准 |
|--------|------|
| 工伤指引 | 完整覆盖 6 个环节，法条引用准确 |
| 免责声明 | 100% 输出带免责标识 |
| 复杂转法务 | 高风险场景自动提示转法务 |
| 次数限制 | 达到配额后提示不可用 |
