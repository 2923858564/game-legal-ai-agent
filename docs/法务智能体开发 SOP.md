# AI 智能体开发 SOP

> 基于 Claude Code 开发验证 → 飞书业务测试 → 目标平台迁移的三层架构。
> 适用于"先做对、再用顺、最后搬"的场景。

---

## 总体流程

```
Phase 0: 知识资产化
    ↓
Phase 1: Claude Code 原型验证
    ↓
Phase 1.5: 功能验收
    ↓ (达标准入)
Phase 2: 飞书Bot接入测试 ←→ 反馈循环（回到Phase 1）
    ↓ (功能稳定)
Phase 3: 功能固化 + 知识库清洗
    ↓
Phase 4: 打包迁移 + 平台适配
    ↓
Phase 5: 回归验证 + 文档
```

---

## Phase 0：知识资产化

**目标**：将业务知识转化为可被 AI 调用的结构化数字资产。

**操作**：
- 收集业务领域的高频知识（法条、合同模板、案例、流程规则）
- 构建结构化的知识库 / skill 文件
- 明确知识的边界：能做什么、不能做什么
- 存入 Claude Code 可调用的位置

**交付物**：结构化知识库 + 技能定义文件

**为什么重要**：这一步决定了后续所有工作的质量上限。知识结构的质量直接决定迁移成本。

---

## Phase 1：Claude Code 原型验证

**目标**：利用 Claude Code 的开发和社区生态优势，快速验证功能可行性。

**操作**：
- 工作流设计（多轮对话 / 单轮处理 / 分级处理）
- 知识库对接（调用哪些 skill、加载策略）
- 输出格式打磨（审核意见书、分析报告、摘要）
- 边界测试（异常输入、边缘场景、批量处理）

**交付物**：
- 可运行的 CLI 原型
- 工作流文档

**注意事项**：
- 定义"完成"的标准，避免无休止调优
- 使用社区现有资源（已有 skill、已有工具链）

---

## Phase 1.5：功能验收

**目标**：在交给业务人员之前建立验收标准，防止过早引入反馈噪音。

**操作**：
- 预设测试场景，检查功能是否能跑通
- 评估输出质量：业务人员能直接使用还是需要大量修改？
- 估算 ROI：处理一项任务的时间缩短了多少？
- 明确不做什么：功能范围的边界

**验收决策**：

| 结果 | 处理 |
|------|------|
| 通过 | 进入 Phase 2 |
| 有条件通过 | 修补问题后进入 Phase 2 |
| 不通过 | 回到 Phase 1 继续开发 |

---

## Phase 2：飞书Bot接入测试

**目标**：将原型接入飞书，让实际业务人员使用和测试。

**接入方式**：
| 方式 | 适用场景 | 特点 |
|------|---------|------|
| cc-connect（Claude Code 直连） | 快速测试、早期验证 | 部署快、沿用 Claude 能力 |
| Python relay（inbox/outbox） | 需要自定义逻辑 | 灵活可控 |
| 飞书原生智能体 | 稳定后迁移目标 | 无服务器、权限可控 |

**反馈管理**：
- 用飞书多维表格结构化收集反馈
- 按优先级分类：bug / 优化 / 新需求
- 反馈循环：收集 → 分类 → 排期 → 修改 → 重新发布

**关键**：反馈不能靠聊天记录，结构化登记才能看到全貌。

---

## Phase 3：功能固化 + 知识库清洗

**目标**：在业务测试获得稳定反馈后，做一次全面清理。

**操作**：
- 工作流精简（测试中有没有冗余步骤？）
- 知识库修正（纠正测试中发现的知识错误）
- 经验沉淀（将教训写入记忆/经验文件）
- 功能裁剪（去掉不需要的功能）

**交付物**：稳定版本 + 清洗后的知识库 + 经验教训记录

---

## Phase 4：打包迁移 + 平台适配

**目标**：将验证过的功能迁移到实际工作场景所用的平台。

**格式映射**：

| Claude Code 环境 | 目标平台（飞书智能体等） |
|-----------------|------------------------|
| skill 文件（.skill / SKILL.md） | 知识库文档 |
| Python 自动化脚本 | 规则引擎 / 人工流程替代 |
| session 文件（多轮状态） | 平台原生会话管理 |
| claude -p 调度指令 | 平台路由规则 / 意图识别 |
| 记忆文件（经验沉淀） | 知识库补充材料 |

**原则**：迁移成本与 Phase 0 的知识结构化程度成反比——知识越规整，迁移越简单。

---

## Phase 5：回归验证 + 文档

**目标**：迁移后验证功能完整性，输出面向业务人员的文档。

**操作**：
1. 准备标准测试用例
2. 迁移前（Claude Code）跑一遍，保存输出
3. 迁移后（目标平台）跑同样的用例
4. 比对输出，分析差异
5. 补充或补偿功能缺口

**业务文档内容**：
- 智能体能做什么 / 不能做什么（功能边界）
- 输入格式要求（怎么提需求）
- 输出规范（如何解读结果）
- 免责声明（不构成正式专业意见）

---

## 核心原则

1. **先验证再投入** — 不要过早投入资源。Phase 1.5 的验收标准是防止方向错误的阀门。
2. **知识先行** — Phase 0 是所有环节的天花板。知识结构的质量决定了最终交付物的质量。
3. **反馈结构化** — 业务人员的反馈必须结构化登记，否则会演化成无底洞。
4. **迁移成本靠前期减** — 迁移是否顺利不取决于 Phase 4，而取决于 Phase 0 把知识组织得多好。
