## 1. 产品概述

### 1.1 项目背景

传奇游戏公司法务部日常处理以下业务场景：

- **合同管理**：推广合作、联运、外包、采购等合同的预审→终审→归档全流程
- **诉讼管理**：传奇IP侵权（私服/外挂/授权链）、合同纠纷、劳动仲裁、知识产权等案件的跟踪与节点管理
- **员工咨询**：纠纷建议、法律知识查询、工商投诉等跨部门法务需求
- **广告合规**：游戏推广素材的合规审查（极限词、虚假宣传、概率公示等）
- **客诉处理**：玩家退款、欺诈举报、未成年人消费等客诉的法务支持

当前流程依赖法务人员手工处理，存在以下痛点：

- 合同审批完成后，归档字段需人工整理至表格，耗时且易遗漏
- 法务重复回答同类咨询（工伤流程、合同预审基础问题）
- 诉讼案件节点分散在个人记录中，缺乏统一跟踪
- 员工侧缺乏自助法务工具，小事也需@法务
- 法律相关的问题一旦出现失误风险很大，法务人员对AI处理的结果不够信任

### 1.2 产品目标

| 目标 | 描述 |
|------|------|
| 合同资产可查可统计 | 审批完成后自动归档至多维表格，结构化可检索。合同流程结束7日后，纸质版盖章版扫描链接栏为空白的，设置定时提醒，提醒法务+合同发起人及时归档合同 |
| 对内核心能力可用 | 终审、诉讼管理、文书起草、法律知识FAQ等法务专用能力可稳定输出 |
| 对员工自助可用 | 合同预审、纠纷路径建议、FAQ等员工可自助完成，复杂场景转法务 |
| 交付可运营可迭代 | 知识库模块化、提示词可维护、法务人员可上手配置 |

### 1.3 建设原则

1. **全程飞书体系**：优先使用飞书原生能力（多维表格、Aily、审批连接器），不单独开发网页版，非技术人员也可轻松使用。
2. **迁移包交付模式**：以法务助手迁移包 + 员工弱化包为交付物，权限和能力隔离。迁移包在本地开发测试后打包，可供飞书智能体或其他AI Agent吸收集成，每个功能模块以测试迭代好的规范输出为标准。
3. **输出可校验**：所有法律分析输出必须符合引用规范（法可依 + 参考例文原文引用）；涉及引用法律条款的，需二次核验（上网搜索检查，确保该法律依据为现行有效法律）。
4. **渐进交付**：Claude Code 原型验证 → 飞书Bot 内测 → 迁移包打包 → 飞书智能体集成上线。

### 3.4 模块 M4：横切能力

#### 引用规范（强制约束，所有智能体通用）

所有法律分析输出必须遵循以下规范（已原型验证通过）：

- **法可依原则**：每条建议必须有法律依据引用且要具体到原文片段
  - 格式：《法律名称》第X条 / 法释〔20XX〕XX号
  - 无法条标注格式：【法律依据】（网上搜索：{来源简述}）
  - 完全未查到格式：【法律依据】（未查到直接引用的法例）
- **参考例文规则**：凡是输出建议/方案/话术，必须附参考例文原文
  - 原文引用，不得改写概括
  - 标注参考来源出处
- **二次核验**：涉及法条引用时，上网搜索确认该法律依据是否为现行有效法律

**自检清单**（输出前检查）：

1. 每条建议是否有法律依据？
2. 参考例文是否原文引用？
3. 引用来源是否标注？
4. 法条是否现行有效（经二次核验）？
5. 法规名称+条文号是否准确？

**条款组织规范**：

- 合同起草/修订输出时，按业务场景分条，不按法律概念拆分（已原型验证通过）

| 示例 | 说明 |
|------|------|
| 正例：第三条 下架处理 → 3.1 责任归属 → 3.2 处理方式 → 3.3 对应费用 |
| 反例：第四条 费用（只写钱）... 第七条 责任（读者来回翻） |

### 3.5 客诉处理专项

| 需求ID | 描述 | 优先级 |
|--------|------|--------|
| CS-01 | 五段式处理流程：类型诊断→案例参照→操作步骤→回复话术→风险防控 | P0 |
| CS-02 | 话术建议必须附参考例文原文 | P0 |
| CS-03 | 游戏行业专项：退款纠纷、充值争议、虚假宣传投诉 | P0 |
